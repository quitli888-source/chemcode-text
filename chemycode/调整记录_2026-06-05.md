## 调整清单005 — 4 项修复

### P1: Minimax 模型 404 Not Found

**原问题**: 配置 Minimax 模型后请求返回 404。

**根因分析**:
1. 模型目录 (`catalog.ts`) 中的模型 ID 使用小写 (`minimax-m3`)，但 MiniMax API 要求精确匹配官方模型名 (`MiniMax-M3`)。
2. 用户按 MiniMax 官方文档设置 `apiUrl` 为 `https://api.minimaxi.com/v1`，LLM 客户端再拼接 `/v1/chat/completions`，导致 URL 变为 `.../v1/v1/chat/completions`，返回 404。

**修复**:

| 文件 | 修改 |
|------|------|
| `server/src/models/catalog.ts` | 模型 ID 改为官方大小写：`MiniMax-M3`、`MiniMax-M2.7`、`MiniMax-M2.5`；新增 `MiniMax-M2`；`findPreset()` 改为大小写不敏感匹配 |
| `server/src/llm/client.ts` | `streamChatCompletion()` 和 `chatCompletion()` 中增加 URL 规范化：`apiUrl.replace(/\/+$/, '').replace(/\/v1$/, '')`，避免双 `/v1` |
| `server/src/routes/models.ts` | 模型测试端点同样增加 URL 规范化 |

**效果**: 用户无论设置 `https://api.minimaxi.com` 还是 `https://api.minimaxi.com/v1`，最终 URL 都正确；模型名使用官方大小写，API 不再返回 404。

### P2: 对话框与 Skills 视图切换丢失数据

**原问题**: 在对话框中点击 Skills 上传 skill，上传成功后切回对话框再切回 Skills，找不到已上传的 skill。

**根因分析**: `skills-view` 组件在切换视图时被销毁重建，但没有在重新挂载时从服务器刷新数据。如果上传在组件销毁期间完成，新组件使用旧的内存状态，看不到新 skill。

**修复**:

| 文件 | 修改 |
|------|------|
| `src/views/skills-view.ts` | `connectedCallback()` 中新增 `refreshSkillsFromServer()` 调用，每次挂载时从服务器获取最新 skills 列表 |

**效果**: 无论用户何时切回 Skills 视图，数据都会从服务器刷新，确保显示最新状态。

### P3: Skill 识别数量异常（1 个 skill 识别出 6 个）

**原问题**: 上传一个 zip 文件，里面只有 1 个 skill，但系统识别出 6 个。

**根因分析**: 多 skill 导入分支 (`findAllManifests` 返回多个结果时) 没有按名称去重。如果同一 zip 被多次导入，或 zip 内存在同名目录，store 中会出现重复条目。

**修复**:

| 文件 | 修改 |
|------|------|
| `server/src/routes/skills.ts` | 多 skill 导入循环中，新增按名称去重逻辑：`data.skills.findIndex((s) => s.name === skillName)` → 存在则先删除旧条目再添加新条目 |

**效果**: 导入时自动去重，同一名称的 skill 不会重复出现在列表中。

### P4: 对话回复中断（模型未返回内容 / 后续回复空白）

**原问题**:
1. ChemCode 找到 skill 后回复莫名其妙中断。
2. 补充问答后返回空白回复。

**根因分析**:
1. LLM 偶尔返回空内容（无 text、无 thinking、无 tool_calls）。
2. Agent 循环检测到空内容后发送重试 `thinking` 事件并 `continue`。
3. 但当循环最终返回时，ws.ts 发送 `done` 事件。
4. 前端收到 `done` 后取消订阅 (`unsub`)。
5. 如果重试产生了内容（`text_delta`），前端已经不在监听，内容丢失。
6. 后续对话的事件也无人接收，导致空白回复。

**修复**:

| 文件 | 修改 |
|------|------|
| `server/src/agent/loop.ts` | 空内容重试不再 `break` 出循环，而是 `continue` 继续重试。只有在获得实际内容或重试次数耗尽时才 `break`。重试耗尽时设置 fallback 内容 `'(模型未返回内容，请重试)'`。 |
| `src/state.ts` | 移除 `done` 事件处理中的冗余空内容检测（已由 agent 循环内部处理）。 |

**关键变化**:
```typescript
// 修复前：空内容重试后 continue，但循环仍可能在下一轮 break
// → ws.ts 发送 done → 前端 unsub → 重试内容丢失

// 修复后：空内容重试后 continue，不 break
// → 循环持续运行直到获得内容 → ws.ts 发送一次 done → 前端收到完整内容
```

**效果**: LLM 返回空内容时自动重试，重试产生的内容不会丢失。用户始终能看到完整回复。
